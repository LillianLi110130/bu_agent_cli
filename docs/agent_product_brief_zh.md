# Agent产品汇报稿

## 1. 产品定位

当前我做的这个产品，不是传统意义上的聊天机器人，而是一个面向研发与知识工作场景的 Agent 工作助手。

它的目标是让大模型从“回答问题”进一步升级到“围绕目标持续做事”。也就是说，用户给出一个任务后，系统不仅能理解意图，还可以结合上下文继续推进任务、调用可用能力、分步骤完成工作，并最终给出结果。

从产品形态上，它已经具备两个特征：

- 一是可以作为本地交互式助手使用，适合研发、自助分析、项目处理等高频工作场景。
- 二是具备向团队级服务化入口演进的基础，可以接入外部消息渠道，承接更连续的任务协作。

## 2. 当前采用的核心思路

这套产品的核心不是单一模型，而是“LLM + Agent + 工具能力 + 多角色协作”的组合方案。

### 2.1 LLM层

底层采用的是兼容 OpenAI 协议的大模型接入方式，目前已经支持在同一套产品框架下灵活切换不同模型配置。当前配置重点使用 GLM 系列模型，也预留了视觉模型能力，意味着系统既能处理文本任务，也具备扩展到图片理解类任务的基础。

对领导视角来说，可以把这一层理解为“通用智能底座”，负责语言理解、推理、归纳和生成。

### 2.2 Agent层

Agent 层负责把单次问答变成连续任务执行。它不只是返回一句答案，而是会根据当前目标判断下一步应该做什么，例如：

- 直接给出答案
- 继续查找信息
- 调用某项能力处理任务
- 把复杂任务拆成多个步骤继续推进

所以这个产品的本质，是把大模型从“对话接口”变成“任务执行主体”。

### 2.3 工具能力层

如果只有大模型而没有工具，系统本质上还是“会说不会做”。当前产品已经给 Agent 配好了可以直接调用的工作能力，让它能真正落地处理任务。

这一层目前重点支持：

- 文件读取、写入、编辑
- 内容搜索与信息定位
- 任务清单与状态跟踪
- 项目资料理解
- 子任务分发与结果回收

### 2.4 多角色协作思路

当前产品不是只做一个“万能助手”，而是已经具备按角色分工处理任务的思路。对于复杂任务，可以交给不同的能力角色分别处理，再统一汇总结果。

这意味着它未来可以进一步从单一 Agent 演进为“多 Agent 协同工作平台”。

## 3. 整体架构

如果从产品架构层面做简化，可以把当前系统分为四层：

### 3.1 交互层

负责承接用户输入和展示结果。

- 支持本地交互式使用
- 支持多轮连续对话
- 支持命令式交互
- 具备接入外部消息渠道的能力

这一层解决的是“用户怎么用”的问题。

### 3.2 智能决策层

这一层由大模型和 Agent 共同完成，负责理解需求、判断路径、组织执行步骤。

它解决的是“系统理解完用户目标之后，下一步做什么”的问题。

### 3.3 能力执行层

这一层承载真正可落地的工作能力。

它解决的是“系统靠什么去完成任务”的问题。

### 3.4 扩展与治理层

这一层负责让产品可持续演进，而不是只能做当前几件事。

- 支持插件式扩展
- 支持角色能力扩展
- 支持会话隔离
- 支持人工审批与使用边界控制
- 支持模型切换

它解决的是“产品能不能稳定落地、能不能持续扩展”的问题。

## 4. 为什么当前优先选择CLI模式

这部分我认为需要单独说明，因为它和产品能力能否真正落地直接相关。

如果只是做一个网页聊天框，那么产品当然更容易展示，也更适合面向更广泛用户做统一入口；但对于当前这个 Agent 产品来说，CLI 不是一个“临时界面”，而是一个更适合研发场景和任务执行场景的产品形态。

原因主要有四点。

### 4.1 CLI更接近真实工作现场

当前产品的核心使用场景，本质上都发生在本地工作目录和研发环境里，例如：

- 阅读项目文件
- 搜索配置和代码
- 修改文档或脚本
- 执行命令
- 跟踪任务状态

CLI 直接就在这些工作对象旁边，Agent 可以更自然地进入真实工作上下文。

相比之下，网页模式更像一个远端会话入口，适合对话和结果展示，但离实际工作现场更远。

### 4.2 CLI模式更适合接入“可执行工具”

当前 Agent 的价值，不只是聊天，而是“调用能力完成任务”。CLI 形态天然更适合承载这类工具能力，例如：

- 文件读写
- 内容搜索
- Bash命令执行
- 任务记录与推进
- 子任务分发

这些能力一旦接入，Agent 就从“给建议”升级为“能落地操作”。这也是 CLI 和普通网页聊天模式最大的差别。

### 4.3 CLI模式更适合研发用户的工作习惯

对于技术团队来说，很多高频工作本来就在终端里发生，例如查日志、看目录、改文件、跑命令、验证结果。CLI 模式不需要用户切出当前环境再去另一个网页里描述问题，使用链路更短，任务切换成本更低。

简单说，网页更适合“问问题”，CLI 更适合“把事情做完”。

### 4.4 CLI更适合作为第一阶段产品形态

从产品演进角度看，CLI 还有一个现实优势，就是它更适合作为第一阶段快速验证 Agent 真正能力的载体。

因为在 CLI 里，我们更容易验证一个关键问题：

**这个 Agent 到底是在回答，还是已经能真正完成任务。**

如果这件事在 CLI 场景里成立，再往网页、团队入口和服务化场景扩展，会更扎实。

## 5. CLI模式相对网页模式的区别

如果做一个简化对比，可以这样理解：

### 5.1 网页模式更擅长的部分

- 门槛更低，适合更广泛用户
- 展示更友好，适合结果呈现和统一入口
- 更适合后续做运营化、平台化和团队共享

### 5.2 CLI模式更擅长的部分

- 更接近本地真实工作对象
- 更容易接入可执行工具
- 更适合复杂任务的连续推进
- 更符合研发用户日常习惯
- 更容易把“分析”直接变成“执行”

所以从当前阶段看，不是网页模式不重要，而是 CLI 模式更适合先把 Agent 的核心价值做透。

## 6. 在CLI里加入文件读写、Bash等工具，具体带来了什么好处

这一点其实是整个产品价值提升的关键。

如果没有这些工具，Agent 更多还是一个增强版问答助手；有了这些工具之后，它才具备“真正参与工作”的能力。

### 6.1 从“描述问题”变成“处理问题”

过去用户需要自己去打开文件、复制内容、执行命令，再把结果贴给模型看。现在 Agent 可以直接处理这些步骤，用户只需要给出目标。

这会把用户交互从“手动串联流程”变成“目标驱动流程”。

### 6.2 从“泛泛建议”变成“基于真实对象工作”

例如在研发场景里，Agent 不再只是泛泛地说“建议检查配置文件”或“可以搜索某个关键词”，而是能够直接去读文件、搜内容、执行检查动作，再根据真实结果继续分析。

这使得输出质量明显更高，也更贴近实际工作。

### 6.3 从“单步回答”变成“连续执行”

当文件读写、搜索、命令执行这些能力被串起来后，Agent 就可以形成完整任务链路，例如：

- 先定位文件
- 再读取内容
- 再搜索相关信息
- 再修改内容
- 再执行验证动作
- 最后汇总结果

这类链路是普通网页聊天模式很难自然承载的。

### 6.4 让结果更可验证

当 Agent 可以调用 Bash 或处理本地文件时，它给出的结论不再只是“模型认为”，而是可以基于实际执行结果来反馈。

这对技术场景尤其重要，因为研发工作非常重视可验证性。

### 6.5 让Agent真正进入工作流

文件读写、命令执行、任务跟踪这些能力，本质上就是把 Agent 嵌入到现有工作流里，而不是让它停留在工作流外部。

这意味着它不只是一个旁观式助手，而是一个能够参与实际生产过程的助手。

## 7. 当前产品已经能做什么

这一部分是最核心的。就当前能力而言，这个 Agent 产品已经不是概念验证，而是具备较强实用性的工作助手。

### 4.1 做持续任务，不只是单轮问答

和普通聊天式 AI 的区别在于，它不是一问一答后就结束，而是可以围绕一个任务持续工作。

例如：

- 用户提出一个目标后，它可以继续追踪任务是否完成
- 能在多轮对话中保持上下文
- 可以在执行过程中继续补充、修正和推进结果

这类能力特别适合研发、分析、整理和协同类工作。

### 4.2 理解项目内容和本地资料

当前产品已经能够围绕本地工作内容开展工作，典型能力包括：

- 理解项目目录和关键文件
- 查找指定信息
- 阅读文档、配置和代码内容
- 对已有资料进行总结和解释

这意味着它不是只回答通用知识问题，而是已经能进入“结合真实工作对象处理任务”的阶段。

### 4.3 支持研发辅助场景

这是当前产品最成熟、最有价值的方向之一。

目前已经可以支持：

- 代码阅读与代码理解
- 根据需求辅助修改代码
- 结合上下文分析报错或问题原因
- 生成和整理技术说明文档
- 辅助开展代码评审

从产品角度看，它已经具备“研发助手”的基础产品形态。

### 4.4 支持任务拆解与进度管理

对于稍复杂一点的工作，系统已经不仅能给建议，还能帮助把工作拆开来做。

当前具备的能力包括：

- 把一个大任务拆成若干步骤
- 记录任务状态
- 展示当前进行到哪一步
- 在阶段完成后汇总结果

这对需求分析、开发任务推进、问题排查和专项处理都比较有价值。

### 4.5 支持多角色/子任务协同

在复杂任务里，当前产品已经具备将工作分发给不同能力角色处理的思路和能力。

这类能力适合处理：

- 需求梳理
- 技术方案设计
- 任务分解
- 代码评审
- 专项分析

简单理解，就是它不只是一个“单线程助手”，而是具备“分工协作”能力的雏形。

### 4.6 支持插件化工作流

除了通用能力之外，当前产品已经可以通过插件方式沉淀专项工作流。现阶段已经覆盖了几类比较明确的场景：

- 需求分析与任务拆解类流程
- 前端工作流类流程
- 代码评审类流程

这件事的价值在于，产品不是每次从零开始处理任务，而是可以逐步沉淀出“标准化工作模板”，让常见任务变成可复用能力。

### 4.7 支持多渠道接入

当前产品除了本地终端使用，还具备接入外部消息渠道的能力。这意味着它未来可以从个人本地助手，逐步演进为团队可共享的智能工作入口。

对产品演进来说，这一点非常关键，因为它决定了后续能否进入团队服务化场景。

### 4.8 支持视觉能力扩展

当前模型配置中已经包含视觉模型预设，因此产品不仅可以处理文字输入，也为图片理解、图文混合分析等能力预留了基础。

这虽然还不是当前主打场景，但对后续扩展文档截图理解、界面分析、图片辅助处理等方向有帮助。

## 8. 当前产品更适合解决什么问题

如果从实际落地角度判断，当前产品最适合先在以下场景形成价值闭环：

- 研发效率提升
- 项目资料理解与知识问答
- 技术任务拆解与推进
- 文档整理和内容产出
- 代码评审与问题排查辅助

一句话概括，当前它最适合处理“需要理解上下文、需要分步骤推进、需要调用实际能力”的知识工作任务。

## 9. 当前阶段判断

我对这个产品当前阶段的判断是：

它已经明显超过了“简单对话助手”阶段，也不仅仅是一个演示性质的 Agent Demo，而是已经具备了比较完整的 Agent 产品雏形。

具体体现在：

- 已经有明确的 LLM + Agent + 工具能力组合方案
- 已经有可持续使用的交互入口
- 已经能处理真实工作对象，而不是只做开放式聊天
- 已经有任务协同、插件扩展和多角色能力的基础

所以当前更适合把它定义为：

**一个面向研发与知识工作场景的智能 Agent 工作平台雏形。**

## 10. 下一步建议

如果接下来以产品化为目标，我建议重点做三件事。

### 7.1 聚焦高价值场景

优先围绕最能体现价值的方向做深，例如：

- 研发助手
- 代码评审助手
- 需求拆解助手
- 项目资料问答助手

先形成几个稳定、可复用的标杆场景。

### 7.2 从“能做”走向“好用”

当前产品能力已经基本成立，下一步重点应该转向体验层面，例如：

- 结果稳定性
- 任务过程可视化
- 使用门槛降低
- 反馈更清晰

### 7.3 持续沉淀标准化工作流

把高频任务沉淀成插件、角色模板和标准执行流程，这样产品价值会从“个人效率工具”逐步升级为“团队知识与任务协作平台”。

## 11. 一句话总结

当前这个 Agent 产品，已经不是单纯调用大模型做问答，而是通过 LLM、Agent、工具能力和多角色协作，把大模型真正变成一个可以持续完成任务的工作助手，并且已经具备继续产品化和平台化演进的基础。
