# Token 配额与上下文窗口分阶段方案

## 1. 背景

当前 CLI 已经具备以下基础能力：

- 模型预设与手动/自动切换：`config/model_presets.json`、`cli/model_switch_service.py`
- 会话上下文预算评估：`agent_core/agent/budget.py`
- 上下文压缩与 working set：`agent_core/agent/compaction/service.py`
- CLI 会话级运行时目录：`cli/session_runtime.py`
- 用户级本地状态根目录：`agent_core/runtime_paths.py`

但目前仍缺少两类面向产品的能力：

1. 用户在 CLI 中无法直观看到“当前会话已经占用了多少上下文窗口”。
2. 系统尚未形成“按用户累计 token 配额治理”的完整闭环。

本方案将该需求拆成三个阶段推进，优先解决可见性，再逐步补足配额控制与云端同步。

## 2. 设计原则

### 2.1 明确区分两类 token 语义

本需求涉及两类不同的 token 数据，必须分开设计：

- 会话上下文 token
  - 用于回答“当前对话离模型上下文窗口还有多远”
  - 主要服务于 CLI 展示、阈值预警、压缩触发可视化
  - 数据来源以本地估算为主
- 用户累计配额 token
  - 用于回答“这个用户本周期还能不能继续用某档模型”
  - 主要服务于额度控制、自动降级、云端对账
  - 数据来源以每次真实 LLM 返回的 usage 为主

如果把这两类数据混在一起，会导致“窗口不够”和“额度用完”两种完全不同的行为原因难以解释，也不利于后续产品化。

### 2.2 先解决可见性，再做治理

当前最紧迫的问题不是立刻做强约束，而是让用户和研发能清楚看到：

- 当前用了什么模型
- 当前会话大概占了多少上下文窗口
- 什么时候接近压缩
- 什么时候正在压缩

因此第一阶段先做“展示与观测”，不引入用户配额控制与云端依赖。

### 2.3 本地事实先稳定，再做云端汇总

由于当前 LLM 接口仍是本地直连，token 事实数据只能先在本地形成稳定闭环。云端同步应建立在本地 usage 事实已经结构化、可追踪、可重放的前提上，而不是反过来驱动本地逻辑。

## 3. 总体分阶段规划

## Phase 1：上下文窗口展示

### 3.1 目标

在 CLI 中持续展示当前会话的上下文窗口占用情况，并在触发压缩时给出明确状态反馈。

### 3.2 本阶段要解决的问题

- 当前模型是什么
- 当前会话估算上下文 token 是多少
- 当前模型上下文窗口大小是多少
- 当前占用百分比是多少
- 当前是否处于压缩中

### 3.3 本阶段范围

本阶段做：

- 会话级上下文 token 估算复用现有 `ContextBudgetEngine`
- CLI 底部状态栏展示
- 压缩状态机与“压缩中”提示
- 会话级快照落盘，便于调试和排障

本阶段不做：

- 用户累计 token 配额
- 配额封顶和限流
- 额度触发的自动切模
- 云端同步

### 3.4 预期产出

- CLI 中始终可见的上下文窗口状态栏
- 可追踪的会话级上下文快照
- 清晰区分“接近压缩”和“正在压缩”

### 3.5 风险与边界

- 展示值主要是本地估算，不等于服务端真实 tokenizer 精确值
- 该阶段目标是“趋势可见、阈值可解释”，不是计费精度

## Phase 2：本地用户配额与自动切模

### 4.1 目标

在不依赖云端的前提下，实现按用户累计 token 的本地配额治理，并在额度达到上限后自动切换到其他模型档位。

### 4.2 本阶段要解决的问题

- 如何标识“当前用户”
- 如何在本地跨会话累计 usage
- 如何定义模型档位和额度规则
- 如何在每轮请求前判断是否还能继续使用当前大模型
- 如何在额度不足时自动降级到其他模型

### 4.3 本阶段范围

本阶段做：

- 本地用户主键抽象
- 本地 usage ledger
- 按模型档位的大模型额度控制
- 配额不足时自动切模
- CLI 展示当前用户在当前档位的已用百分比

本阶段不做：

- 云端统一汇总
- 多设备强一致额度
- 服务端仲裁

### 4.4 核心设计方向

- 用户标识
  - 本地 CLI 单机模式先用本地默认用户标识
  - worker 模式复用 `cli/worker/auth.py` 已持久化的 `user_id`
- 配额数据结构
  - 引入本地持久化 usage ledger
  - 建议按用户、档位、周期聚合
- 模型治理方式
  - 不直接按 `model name` 控制，而是引入 `large / standard / economy` 档位
  - 档位之间维护降级链
- 路由位置
  - 在每轮请求前做准入判断
  - 如果当前档位额度不足，则调用现有 `ModelSwitchService` 自动切换到 fallback preset

### 4.5 预期产出

- 本地单机可用的用户配额体系
- large 模型额度耗尽后的自动降级
- CLI 中可视化的用户档位配额占比

### 4.6 风险与边界

- 本地账本天然不具备跨设备强一致性
- 用户删除本地状态文件可能绕过约束
- 因此该阶段只能视为“客户端治理”，不应视为最终风控方案

## Phase 3：云端同步与统一配额

### 5.1 目标

将本地真实 usage 事件同步到云端，形成跨设备统一的用户配额、审计与统计体系。

### 5.2 本阶段要解决的问题

- 如何将本地 usage 事实以可幂等方式上报云端
- 如何在多设备之间统一用户额度
- 如何离线使用后补传
- 如何让云端额度与本地路由策略协同

### 5.3 本阶段范围

本阶段做：

- usage event 增量同步
- 云端幂等去重
- 云端聚合与报表
- 云端策略下发
- 本地与云端对账

### 5.4 核心设计方向

- 同步粒度使用“事件”，而不是只上传汇总值
- 每条 usage event 具备唯一 event_id，服务端幂等入库
- 本地维护 `pending_sync / synced / failed` 状态
- 在线时优先使用云端策略，离线时回退本地缓存策略

### 5.5 预期产出

- 多设备统一的用户额度治理
- 服务端审计与报表能力
- 本地离线后可恢复补传

### 5.6 风险与边界

- 需要定义新的服务端接口和数据模型
- 需要处理事件补传、乱序、去重和策略缓存

## 4. 推荐推进顺序

建议按以下节奏推进：

1. 先完成 Phase 1
   - 目标是把“当前上下文窗口状态”稳定展示出来
   - 这是最小改动、最高感知价值的一步
2. 再完成 Phase 2
   - 在本地形成用户配额治理闭环
   - 这一步复用 Phase 1 的展示框架
3. 最后完成 Phase 3
   - 将本地 usage 事实接入云端汇总、统一配额与审计

## 5. 阶段依赖关系

- Phase 1 是基础可视化层，为 Phase 2 和 Phase 3 提供统一展示入口。
- Phase 2 依赖本地 usage 事件稳定落盘，但不依赖云端。
- Phase 3 依赖 Phase 2 已经形成稳定的本地 usage ledger 结构。

## 6. 当前结论

本需求最合理的落地路径不是一次性实现“展示 + 本地配额 + 云端同步”，而是：

1. 先把会话上下文窗口做成稳定、可信、可观察的 CLI 能力。
2. 再把本地用户配额治理接上。
3. 最后再把本地 usage 事实同步到云端，形成统一额度系统。

该拆分方式可以最大程度复用现有代码中的预算评估、压缩、模型切换和运行时目录能力，同时降低第一阶段改动面与验证成本。
